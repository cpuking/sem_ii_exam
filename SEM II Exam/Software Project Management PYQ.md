## Q2a) Explain the Capability Maturity Model (CMM) in Detail `[4 marks]`

### Definition

**CMM (Capability Maturity Model)** is a framework developed by the **Software Engineering Institute (SEI), Carnegie Mellon University** that describes the maturity levels of an organization's software development processes. It helps organizations improve their processes systematically.

### Five Levels of CMM

|Level|Name|Description|Characteristics|
|---|---|---|---|
|**1**|**Initial**|No defined processes; ad hoc and chaotic|Unpredictable outcomes; success depends on individual heroics|
|**2**|**Managed (Repeatable)**|Basic project management in place|Projects are planned, tracked; repeatable for similar projects|
|**3**|**Defined**|Standard, documented processes across the organization|Consistent processes; training and process documentation|
|**4**|**Quantitatively Managed**|Processes are measured using statistical methods|Predictable quality; quantitative targets set and met|
|**5**|**Optimizing**|Continuous improvement; defect prevention|Proactive change management; constant process refinement|

### CMM Pyramid Diagram

```
      ╔══════════════════╗
      ║   5. OPTIMIZING  ║  ← Continuous improvement
      ╠══════════════════╣
      ║ 4. QUANTITATIVELY║  ← Measured, predictable
      ║    MANAGED       ║
      ╠══════════════════╣
      ║   3. DEFINED     ║  ← Standardized org-wide
      ╠══════════════════╣
      ║   2. MANAGED     ║  ← Project-level control
      ╠══════════════════╣
      ║   1. INITIAL     ║  ← Chaotic, ad hoc
      ╚══════════════════╝
```

### Benefits of CMM

- Provides a clear roadmap for process improvement
- Helps organizations benchmark their maturity
- Reduces project failures by improving predictability
- Widely adopted globally in software organizations

> ✍️ **Exam tip:** Draw the pyramid + write all 5 levels with one-line description = guaranteed full marks (4/4).

## Q3a) Explain the Processes Involved in Scope Management `[4 marks]`

### Definition

**Scope Management** ensures that the project includes all the work required — and ONLY the work required — to complete the project successfully.

### Processes in Scope Management

|#|Process|Description|Output|
|---|---|---|---|
|**1**|**Scope Planning**|Define how scope will be managed throughout the project|Scope Management Plan|
|**2**|**Scope Definition**|Develop detailed project scope statement — list of deliverables and work|Project Scope Statement|
|**3**|**Create WBS**|Decompose deliverables into smaller, manageable work packages|WBS and WBS Dictionary|
|**4**|**Scope Verification**|Formal acceptance of deliverables by stakeholders|Accepted deliverables|
|**5**|**Scope Control**|Monitor scope and manage changes to baseline|Change requests, updates|

### Work Breakdown Structure (WBS) Example

```
E-Commerce App (Level 0)
├── 1. Frontend         (Level 1)
│   ├── 1.1 Login page
│   ├── 1.2 Product listing
│   └── 1.3 Cart
├── 2. Backend          (Level 1)
│   ├── 2.1 User API
│   └── 2.2 Payment API
└── 3. Testing          (Level 1)
    ├── 3.1 Unit tests
    └── 3.2 UAT
```

### Scope Creep

**Scope Creep** = unauthorized, gradual expansion of project scope without schedule/budget adjustment.

- Prevention: formal change control process, documented scope baseline

## Q3b) Explain Cost Estimation and Planning `[3 marks]`

### Definition

**Cost Estimation** is the process of predicting the financial resources needed to complete a project. **Cost Planning** involves allocating that budget across activities and setting a cost baseline.

### Cost Management Processes

|Process|Description|Output|
|---|---|---|
|**Cost Estimation**|Predict cost of each activity/work package|Cost estimates|
|**Cost Budgeting**|Aggregate estimates into overall budget|Cost baseline (BAC)|
|**Cost Control**|Monitor expenditure, manage variances|Forecasts, change requests|

### Estimation Techniques

|Technique|Description|Accuracy|When Used|
|---|---|---|---|
|**Analogous**|Based on cost of similar past project|±25–50%|Early planning|
|**Bottom-up**|Estimate each task individually, then sum|±5–10%|Detailed planning|
|**Parametric**|Use statistical model (e.g., cost per function point)|±10–20%|When data exists|
|**Three-Point (PERT)**|`(Optimistic + 4×Most Likely + Pessimistic) / 6`|±15%|Uncertain tasks|
|**Expert Judgment**|Experienced person's estimate|Varies|Complex tasks|

### PERT Formula Example

```
O = 4 days, M = 6 days, P = 14 days
Expected cost = (4 + 4×6 + 14) / 6 = 42/6 = 7 days (or units of cost)
```

> ✍️ **Exam tip:** Definition + 3 processes + techniques table + PERT formula = 3/3 marks.

## Q4a) Describe Quality Assurance and Quality Control in Project Management `[4 marks]`

### Definitions

**Quality Assurance (QA):** A proactive, process-oriented approach that focuses on **preventing defects** by ensuring that the processes used to create the product are correct and followed.

**Quality Control (QC):** A reactive, product-oriented approach that focuses on **detecting defects** in the actual deliverables by inspection and testing.

### QA vs QC — Full Comparison Table

|Feature|Quality Assurance (QA)|Quality Control (QC)|
|---|---|---|
|**Orientation**|Process-oriented|Product-oriented|
|**Goal**|Prevent defects|Detect defects|
|**Approach**|Proactive|Reactive|
|**When**|Throughout project lifecycle|After product is built/tested|
|**Who performs**|QA engineers, auditors|QC testers, inspectors|
|**Activities**|Process audits, code standards, training|Testing, inspection, reviews|
|**Output**|Process improvement plans|Defect reports, rework requests|
|**Example**|"Is the team following coding standards?"|"Does this feature pass all test cases?"|
|**Focus area**|How we work|What we produce|
|**Standard**|ISO 9001, CMMI|Test plans, checklists|

### Simple Analogy

> QA = Installing a safety culture in a factory (prevent accidents) QC = Inspecting each product at the end of the line (catch defects)

### Quality Tools Used

|For QA|For QC|
|---|---|
|Process audits|Test cases|
|Checklists|Code reviews|
|Training programs|Bug tracking tools|
|Benchmarking|Acceptance testing|

> ✍️ **Exam tip:** Both definitions (1 line each) + full comparison table + analogy + one tools table = 4/4 marks.

## Q4b) Explain Solicitation and Solicitation Planning `[3 marks]`

### Definition

**Solicitation** is the process of obtaining responses (bids, proposals, quotations) from prospective sellers/vendors to fulfill project procurement needs.

**Solicitation Planning** is the process of preparing the documents needed to support solicitation — deciding what to procure, when, and how.

### Solicitation Planning Documents

|Document|Full Form|Purpose|
|---|---|---|
|**RFP**|Request for Proposal|Detailed requirements; vendors propose solutions|
|**RFQ**|Request for Quotation|Ask vendors to quote price for specified work|
|**IFB**|Invitation for Bid|Formal competitive bidding process|
|**RFI**|Request for Information|Gather market information before formal procurement|

### Solicitation Process Steps

|Step|Description|
|---|---|
|1. **Identify need**|Determine what goods/services need to be procured|
|2. **Prepare documents**|Create RFP, RFQ, or IFB with specifications|
|3. **Advertise**|Publish opportunity to potential vendors|
|4. **Receive responses**|Collect proposals or bids from vendors|
|5. **Evaluate**|Compare against criteria (cost, quality, timeline)|
|6. **Select vendor**|Award contract to best-fit vendor|

### Source Selection Criteria

- Technical capability
- Past experience and references
- Cost/price
- Management approach
- Financial stability

## Q4c) Explain the Scope of Software Metrics `[3 marks]`

### Definition

**Software Metrics** are quantitative measures used to evaluate attributes of a software product, process, or project. The scope of metrics covers everything from code quality to team productivity.

### Scope of Software Metrics

|Scope Area|What It Measures|Example Metric|
|---|---|---|
|**Size measurement**|How large is the software|Lines of Code (LOC), Function Points|
|**Quality assessment**|How defect-free is the software|Defect density = Defects / KLOC|
|**Productivity tracking**|How efficient is the team|LOC / person-month|
|**Cost estimation**|What will the software cost|Cost per function point|
|**Process efficiency**|How good are our development processes|Defect removal efficiency|
|**Reliability**|How often does the system fail|Mean Time Between Failures (MTBF)|
|**Maintainability**|How easy is it to modify|Cyclomatic complexity|
|**Schedule tracking**|Are we on time|Schedule variance (SV = EV - PV)|

### Types of Software Metrics

|Type|Focus|Examples|
|---|---|---|
|**Product metrics**|Properties of the software|Size, complexity, reliability|
|**Process metrics**|Development process quality|Review effectiveness, defect removal rate|
|**Project metrics**|Project management effectiveness|Cost variance, schedule variance|

### Key Formulas to Remember

```
Defect Density      = Number of Defects / KLOC
Productivity        = LOC / Person-month
Test Coverage       = Lines tested / Total lines × 100
Cost Performance    = EV / AC  (CPI > 1 = under budget)
```

## Q5a) Differentiate Between PSP and TSP `[4 marks]`

### Definitions

**PSP (Personal Software Process):** A structured framework developed by **Watts Humphrey (SEI)** for individual software engineers to improve their personal software development practices — focusing on planning, measurement, and quality at the individual level.

**TSP (Team Software Process):** A framework also developed by Humphrey that helps software teams build and manage high-performance teams, using data-driven methods for planning, tracking, and quality management at the team level.

> PSP + TSP together form a complete improvement framework: individuals improve via PSP → teams improve via TSP.

### PSP vs TSP — Detailed Comparison

|Feature|PSP|TSP|
|---|---|---|
|**Level**|Individual developer|Software team|
|**Focus**|Personal work habits and productivity|Team coordination and delivery|
|**Goal**|Write better code with fewer bugs, accurate estimates|Predictable, high-quality team delivery|
|**Planning**|Personal schedule and size estimates|Team project plan (aggregated from PSP data)|
|**Metrics**|Individual LOC, defects, time per task|Team velocity, defect rates, burn-down|
|**Reviews**|Personal self-reviews, design reviews|Peer reviews, team inspections|
|**Foundation**|Standalone; prerequisite for TSP|Built on top of PSP skills|
|**Data tracking**|Developer tracks own time and defects|PM aggregates team data|
|**Roles**|Developer|Team Leader, Team, Coach|
|**Developed for**|Entry-level to senior engineers|Software development teams|

### PSP Levels (for 4-mark depth)

|PSP Level|Name|Focus|
|---|---|---|
|PSP 0|Baseline|Measure current performance|
|PSP 1|Personal Planning|Size estimation, schedule planning|
|PSP 2|Design and Code Reviews|Defect prevention|
|PSP 3|Cyclic Development|Iterative personal process|

### How They Work Together

```
Developer learns PSP
      ↓
Better personal estimation + defect prevention
      ↓
Joins team using TSP
      ↓
Team aggregates PSP data for planning
      ↓
High-performance, predictable project delivery
```

> ✍️ **Exam tip:** Both definitions + full comparison table (8–10 rows) + relationship diagram = 4/4 marks.

## Q5b) Outline the Importance of Risk Management in Project Procurement `[3 marks]`

### Definition

**Risk Management in Project Procurement** involves identifying, analyzing, and responding to potential threats and uncertainties that could affect the acquisition of goods, services, or results from external vendors.

### Why Risk Management is Important in Procurement

|Importance|Explanation|
|---|---|
|**Identifies procurement risks early**|Vendor failure, price changes, delivery delays caught before they occur|
|**Reduces project failure**|Proactive mitigation prevents small risks from becoming crises|
|**Ensures vendor reliability**|Systematic evaluation of vendor capabilities and financial stability|
|**Protects budget**|Contingency reserves set aside for cost overruns from vendor changes|
|**Manages contractual risks**|Identifies unclear contract terms that could lead to disputes|
|**Improves planning accuracy**|Risk-adjusted estimates are more realistic|
|**Supports decision-making**|Risk scores guide make-or-buy decisions|
|**Ensures continuity**|Backup vendor plans prevent single-vendor dependency|

### Common Procurement Risks and Responses

|Risk|Strategy|Response|
|---|---|---|
|Vendor goes out of business|Transfer|Multi-vendor contracts, escrow|
|Delayed delivery|Mitigate|Penalty clauses in contract|
|Cost overrun from vendor|Accept|Budget contingency reserve|
|Poor quality deliverable|Avoid|Strong SLAs and acceptance criteria|
|Vendor lock-in|Mitigate|Open standards, portability clauses|

### Risk Management Process (in Procurement Context)

```
Identify procurement risks → Analyze (Probability × Impact) →
Plan response → Implement → Monitor vendor performance
```

> ✍️ **Exam tip:** Definition + importance table (5–6 points) + risks & responses table = 3/3 marks.

## Q5c) Discuss Why Effective Communication is Crucial for Project Success `[3 marks]`

### Definition

**Project Communication Management** ensures timely, accurate, and appropriate generation, collection, distribution, and disposition of project information to all stakeholders.

### Why Effective Communication is Crucial

|Reason|Explanation|
|---|---|
|**Clarity of goals**|Everyone understands what needs to be done and why|
|**Team coordination**|Tasks are synchronized; no duplicate or missed work|
|**Stakeholder alignment**|Expectations are managed; no surprises at delivery|
|**Early problem detection**|Issues surfaced quickly through regular reporting|
|**Conflict reduction**|Transparent communication reduces misunderstandings|
|**Decision quality**|PM has complete information to make good decisions|
|**Morale and motivation**|Team feels valued when kept informed|
|**Change management**|Changes are communicated properly; fewer disruptions|

### Statistics That Make This Real

> A **PM spends ~90% of their time communicating.** Most project failures are caused by communication breakdown, not technical failure.

### Communication Channels Formula

```
Channels = n(n-1)/2

5 stakeholders  → 10 channels
10 stakeholders → 45 channels
15 stakeholders → 105 channels
```

As team size grows, communication complexity grows **exponentially** — showing why a structured communication plan is essential.

### Communication Plan Elements

|Element|Description|
|---|---|
|**Who**|Which stakeholders need information|
|**What**|Type of information (status, risk, decision)|
|**When**|Frequency (daily, weekly, milestone)|
|**How**|Channel (email, meeting, dashboard)|
|**Format**|Detail level (summary vs full report)|
