## ⭐ Q. Define Project Management `2M` `REPEATED`

**Definition:** Project management is the application of **knowledge, skills, tools, and techniques** to project activities in order to meet project requirements within the constraints of **scope, time, and cost**.

**Key aspects:**

- Planning, executing, and controlling project activities
- Achieving specific goals within a defined timeline and budget
- A project manager oversees all stages from initiation to closure
- Ensures quality, scope, and stakeholder satisfaction

## ⭐ Q. Define WBS and Give Its Purpose `2M`

**Definition:** A **Work Breakdown Structure (WBS)** is a hierarchical decomposition of a project's total scope into smaller, manageable components called **work packages**.

**Purpose:**

- Breaks complex project into manageable tasks
- Provides a clear picture of all deliverables
- Basis for scheduling, cost estimation, and resource assignment
- Prevents scope creep by defining boundaries

**Mini Example:**

```
Website Project
├── Frontend
│   ├── Homepage
│   └── Login page
├── Backend
│   ├── User API
│   └── Database
└── Testing
    └── UAT
```

## ⭐ Q. Define Scope Planning `2M` `REPEATED`

## ⭐ Q. Define Quality Assurance in Context of Project Management `2M` `REPEATED`

**Definition:** **Quality Assurance (QA)** is a **process-oriented** approach that ensures the team is following the defined processes and standards to **prevent defects** throughout the project lifecycle.

**Key Points:**

- Focuses on the _process_, not the product
- Proactive — catches problems before they occur
- Activities: audits, process reviews, training, standards compliance
- Output: Process improvement recommendations

**QA vs QC (quick distinction):**

|QA|QC|
|---|---|
|Process-oriented|Product-oriented|
|Prevents defects|Detects defects|
|Proactive|Reactive|

## ⭐ Q. Write a Note on Software Metrics `3–4M`

### Definition

**Software Metrics** are quantitative measures used to evaluate attributes of software products, processes, and projects. They provide objective data to support decision-making.

### Why Metrics Matter

|Purpose|Example|
|---|---|
|Measure quality|Defect density: 2 bugs per KLOC|
|Track productivity|LOC per person-month|
|Estimate cost/effort|Function Points → cost|
|Monitor progress|Schedule variance, burn-down charts|
|Improve processes|Trend analysis on defect rates|

### Types of Software Metrics

|Type|Focus|Examples|
|---|---|---|
|**Product metrics**|Attributes of the software|Size (LOC), complexity, reliability|
|**Process metrics**|Development process quality|Defect removal efficiency, review rate|
|**Project metrics**|PM effectiveness|Cost variance, schedule variance|

### Key Formulas

```
Productivity    = LOC / Person-month
Defect Density  = Number of Defects / KLOC
Test Coverage   = Lines Tested / Total Lines × 100
Cost per LOC    = Total Cost / LOC
```

---

## ⭐ Q. Explain Size-Oriented Metrics `2–3M` `REPEATED`

### Definition

**Size-oriented metrics** measure software size using **Lines of Code (LOC)** as the primary unit. Other metrics are derived from this base measure.

### Key Metrics

|Metric|Formula|
|---|---|
|Productivity|LOC / Person-month|
|Defects per KLOC|Number of Defects / KLOC|
|Cost per LOC|Total Cost / LOC|
|Documentation ratio|Doc pages / KLOC|

### Advantages vs Disadvantages

|Advantages|Disadvantages|
|---|---|
|Simple to count|Language-dependent (Python vs Java)|
|Objective, universally understood|Doesn't measure quality|
|Historical data widely available|Can encourage "code padding"|
|Easy to automate collection|Ambiguity around blank/comment lines|

---

## ⭐ Q. Explain Function-Oriented Metrics `2–3M` `REPEATED`

### Definition

**Function-oriented metrics** measure software **functionality** (what it does for users) rather than code size. The primary measure is **Function Points (FP)**.

### Function Point Components

|Component|Abbreviation|Description|Example|
|---|---|---|---|
|External Inputs|EI|Data entering the system|Login form, search|
|External Outputs|EO|Data leaving the system|Reports, screens|
|External Inquiries|EQ|User queries to system|Database lookups|
|Internal Logical Files|ILF|Internal data stores|Database tables|
|External Interface Files|EIF|External data sources|APIs, third-party data|

### Advantages vs Disadvantages

|Advantages|Disadvantages|
|---|---|
|Language-independent|Complex to calculate|
|Estimated early (from requirements)|Requires expertise|
|Better for cross-project comparisons|Subjective component classification|
|Captures functional complexity|Not intuitive for developers|

## ⭐ Q. Explain Software Metrics Data Collection and Analysis `3–4M`

### Data Collection

**Definition:** The process of systematically gathering quantitative information about software development activities.

**Types of Data Collected:**

|Data Type|Example|
|---|---|
|Effort data|Person-hours per task|
|Cost data|Budget spent per phase|
|Defect data|Bugs found per KLOC|
|Size data|LOC, Function Points|
|Productivity data|LOC per person-month|

**Sources:** Version control systems (Git), bug tracking (Jira), PM tools (MS Project), developer logs.

**Collection Steps:**

1. Define which metrics to collect
2. Identify data sources
3. Collect data regularly (weekly/per-sprint)
4. Store in a central repository
5. Validate accuracy before analysis

### Data Analysis Methods

|Method|Purpose|Example|
|---|---|---|
|**Trend Analysis**|Identify patterns over time|Rising defect rate over 3 sprints|
|**Comparative Analysis**|Compare across projects|Defect density: Project A vs B|
|**Statistical Analysis**|Mean, median, std deviation|Average LOC per developer|
|**Root Cause Analysis**|Find why problems occur|Why Module X has 3× more defects|

## ⭐ Q. Explain Quality Planning and Assurance, and Their Importance `3–4M` `REPEATED`

### Quality Planning

**Definition:** The process of identifying relevant quality standards for the project and determining how to meet them.

**Key Steps:**

1. Define quality requirements (based on customer needs + industry standards)
2. Plan QA activities (audits, process assessments)
3. Define quality metrics (defect rates, satisfaction scores)
4. Design standard processes
5. Implement reviews and audits

**Output:** Quality Management Plan (QMP)

**Tools:**

- Six Sigma (reduce variation)
- Benchmarking (compare with best practices)
- Control Charts (monitor process stability)
- QFD — Quality Function Deployment

### Quality Assurance (QA) vs Quality Control (QC)

|Feature|QA|QC|
|---|---|---|
|Orientation|Process|Product|
|Goal|Prevent defects|Detect defects|
|When|During execution|After product built|
|Example|Auditing coding standards|Running test cases|

### Importance

|Importance|Reason|
|---|---|
|**Customer satisfaction**|Product meets expectations|
|**Cost savings**|Catching issues early is cheaper than fixing later|
|**Reputation**|Consistent quality builds trust|
|**Compliance**|Meets industry/regulatory standards|

## ⭐ Q. Explain the Staff Acquisition Process `3M` `PYQ REPEATED`

### Definition

**Staff Acquisition** is the process of identifying, recruiting, selecting, and onboarding the right human resources needed to complete the project.

### Steps in Staff Acquisition

|Step|Description|
|---|---|
|**1. Identify Requirements**|Determine skills, roles, number of people needed (from WBS)|
|**2. Human Resource Planning**|Assess staffing needs vs. organizational capacity|
|**3. Recruitment**|Source candidates — internal, external, contractors|
|**4. Selection**|Evaluate via interviews, tests, background checks|
|**5. Onboarding**|Project orientation, tool access, role briefing|
|**6. Training & Development**|Fill skill gaps through OJT or formal training|
|**7. Release Planning**|Plan reassignment when project role ends|

### Key Techniques

- **Pre-assignment** — staff identified before project starts
- **Negotiation** — negotiate with functional managers for resources
- **Virtual teams** — geographically distributed teams

### Brooks' Law (Important Concept)

> _"Adding manpower to a late software project makes it later."_ — Fred Brooks

- New team members need onboarding time
- Adds communication overhead
- Can actually slow the project down

## ⭐ Q. Explain the Capability Maturity Model (CMM) in Detail `4M` `PYQ REPEATED`

### Definition

**CMM** is a process improvement framework developed by **SEI (Software Engineering Institute), Carnegie Mellon University** that defines five levels of organizational process maturity for software development.

### Five CMM Levels

|Level|Name|Description|Key Characteristic|
|---|---|---|---|
|**1**|**Initial**|No defined processes; ad hoc and chaotic|Success = individual heroics|
|**2**|**Managed**|Basic project management in place|Project-level planning and tracking|
|**3**|**Defined**|Org-wide standardized, documented processes|Consistent processes across all projects|
|**4**|**Quantitatively Managed**|Processes measured using statistical methods|Data-driven decisions, predictable quality|
|**5**|**Optimizing**|Continuous improvement; defect prevention|Proactive innovation, constantly improving|

### CMM Pyramid Diagram _(Draw this in exam)_

```
        ╔══════════════════════╗
        ║   5 – OPTIMIZING     ║  Continuous improvement
        ╠══════════════════════╣
        ║  4 – QUANTITATIVELY  ║  Measured & predictable
        ║      MANAGED         ║
        ╠══════════════════════╣
        ║   3 – DEFINED        ║  Org-wide standard processes
        ╠══════════════════════╣
        ║   2 – MANAGED        ║  Project-level control
        ╠══════════════════════╣
        ║   1 – INITIAL        ║  Chaotic, ad hoc
        ╚══════════════════════╝
```

### How to Apply CMM Levels (Organization Use)

|Level|What Organization Does|
|---|---|
|L1 → L2|Introduce project planning, tracking, requirements management|
|L2 → L3|Standardize processes org-wide; create process documentation|
|L3 → L4|Collect metrics; use statistical process control|
|L4 → L5|Use data to drive continuous improvement programs|

### Benefits

- Better predictability of project outcomes
- Reduced rework and defect costs
- Higher client satisfaction
- Competitive advantage in process maturity

### Real-World Example: Raytheon Experience

- Problem: Unpredictable schedules, high defect rates
- Solution: Implemented CMM Level 3 practices
- Result: 2:1 ROI; rework cost dropped from 40% → under 10% of budget

## ⭐ Q. Explain PSP and TSP / Differentiate Between PSP and TSP `4M` `PYQ`

### PSP – Personal Software Process

**Developer by:** Watts Humphrey (SEI) **Focus:** Individual engineer **Goal:** Teach engineers to manage their own work quality, estimate accurately, and reduce defects at the source.

**PSP Levels:**

|Level|Focus|
|---|---|
|PSP0|Baseline — record time and defects|
|PSP1|Planning — size estimation and scheduling|
|PSP2|Quality — design and code reviews before compile|
|PSP3|Cyclic development for larger programs|

**Key PSP Metrics:**

- **Yield:** % of defects found before first compile
- **Cost of Quality (CoQ):** Time on reviews vs time on debugging

### TSP – Team Software Process

**Focus:** Project team (3–20 people) **Goal:** Build and manage self-directed, high-performance software teams.

**TSP Roles:**

- Team Leader — manages schedule
- Development Manager — leads design/implementation
- Planning Manager — tracks estimates
- Quality Manager — monitors defects
- Support Manager — handles tools and config

**TSP Cycle:**

1. Launch (goals + roles)
2. Strategy (how to build)
3. Plan (estimates + schedule)
4. Requirements
5. Design / Implement / Test
6. Post-Mortem (analyze data, improve)

### PSP vs TSP Comparison

|Feature|PSP|TSP|
|---|---|---|
|**Level**|Individual developer|Project team|
|**Focus**|Personal habits, accuracy|Team coordination, delivery|
|**Planning**|Personal schedule + size estimates|Team project plan|
|**Reviews**|Self-reviews|Peer and team inspections|
|**Metrics**|LOC, time per phase, defect injection rate|Team velocity, system defects|
|**Foundation**|Standalone|Requires PSP-trained engineers|
|**Key Motto**|"Know thyself"|"Build the right team"|

### The Pyramid Relationship

```
        CMMI / KPAs   ← Organization process maturity
            │
           TSP        ← Team-level discipline and coordination
            │
           PSP        ← Individual engineer discipline
```

## ⭐ Q. Explain the Information Distribution in Project Management `3M` `PYQ REPEATED`

### Definition

**Information Distribution** is the process of making relevant project information available to the right stakeholders, at the right time, in the right format.

### Communication Methods

|Method|Type|Best For|
|---|---|---|
|**Meetings / Briefings**|Interactive (2-way)|Complex decisions, problem-solving|
|**Email / Reports**|Push (1-way directed)|Status updates, documentation|
|**Project Dashboard / Wiki**|Pull (on-demand)|Real-time progress tracking|
|**Presentations**|Interactive|Milestone reviews, stakeholder briefings|
|**Shared repository (SharePoint)**|Pull|Reference documents, file sharing|

### What to Distribute

|Information Type|Audience|Frequency|
|---|---|---|
|Status reports|Sponsor, management|Weekly|
|Progress updates|Project team|Daily|
|Risk alerts|PM, sponsor|As needed|
|Decisions made|All stakeholders|Immediately|
|Lessons learned|All, archives|End of project|

### Challenge: Information Overload

> Sending a 50-page technical manual to a CEO is as bad as sending nothing. Information must be tailored by audience, format, and detail level.

### Communication Channels Formula

```
Number of channels = n(n-1)/2

5 team members   → 10 channels
10 team members  → 45 channels
15 team members  → 105 channels
```

More people = exponentially more communication complexity.

## ⭐ Q. Discuss Importance of Change Control in Managing Scope, Time, and Cost `3–4M`

### Definition

**Change Control** is the formal process of reviewing, approving, and managing all changes to a project's scope, schedule, or budget to prevent uncontrolled modifications.

### Why Change Control Matters

|Without Change Control|With Change Control|
|---|---|
|Unauthorized scope additions|All changes formally approved|
|Budget and schedule overruns|Impact assessed before approval|
|Conflicting stakeholder expectations|Single source of truth for changes|
|Scope creep|Scope baseline protected|

### Change Control Process

```
Change Request Raised
        ↓
Impact Analysis (scope + time + cost)
        ↓
Change Control Board (CCB) Review
        ↓
Approved? ──YES──► Implement + Update Plan
              |
             NO
              ↓
         Reject / Defer
```

### Triple Constraint (Iron Triangle)

```
           SCOPE
          ╱     ╲
         ╱ QUALITY╲
        ╱  (center) ╲
     TIME ─────────── COST
```

- Increase scope → Increase time AND/OR cost
- Reduce time → Increase cost OR reduce scope
- Reduce budget → Reduce scope OR extend timeline

## ⭐ Q. Identify Stakeholders and Explain How Stakeholder Identification Positively Influences Project Success `3–4M`

### Definition

A **stakeholder** is any person, group, or organization that can affect — or is affected by — a software project.

### Types of Stakeholders

|Type|Examples|
|---|---|
|**Internal**|Project manager, developers, testers, business analysts|
|**External**|Clients, end users, suppliers, government regulators, investors|
|**Primary**|Directly affected (sponsors, end users)|
|**Secondary**|Indirectly affected (media, community)|

### Steps to Identify Stakeholders

1. Review project charter and requirements documents
2. Brainstorm with team members
3. Analyze organizational structure (which departments are involved?)
4. Consult domain experts
5. Create **Stakeholder Register**

### Power-Interest Grid

|Quadrant|Strategy|Example|
|---|---|---|
|High Power, High Interest|**Manage Closely**|Sponsor, Client|
|High Power, Low Interest|**Keep Satisfied**|Senior management|
|Low Power, High Interest|**Keep Informed**|End users|
|Low Power, Low Interest|**Monitor**|General public|

### How It Positively Influences Project Success

|Benefit|Explanation|
|---|---|
|**Clear requirements**|Stakeholders define what they actually need|
|**Fewer surprises**|Known expectations = fewer late-stage conflicts|
|**Better support**|Engaged stakeholders champion the project|
|**Risk reduction**|Early identification of opposing stakeholders|
|**Aligned goals**|All parties working toward same outcome|

## ⭐ Q. Explain Cost Estimation and Planning `3M` `PYQ`

### Definition

**Cost Estimation** predicts the financial resources needed. **Cost Planning** allocates that budget across tasks and sets the cost baseline.

### Estimation Techniques

|Technique|Description|Accuracy|When Used|
|---|---|---|---|
|**Analogous**|Based on similar past projects|±25–50%|Early planning|
|**Bottom-up**|Estimate each task, sum up|±5–10%|Detailed planning|
|**Parametric**|Statistical model (cost/FP or cost/LOC)|±10–20%|When historical data exists|
|**Three-Point (PERT)**|`(O + 4M + P) / 6`|±15%|Uncertain tasks|
|**Expert Judgment**|Experienced person's estimate|Varies|Complex tasks|

### PERT Formula

```
Expected Duration = (Optimistic + 4 × Most Likely + Pessimistic) / 6

Example:
O = 4 days, M = 6 days, P = 14 days
E = (4 + 24 + 14) / 6 = 42/6 = 7 days
```

### EVM (Earned Value Management) Key Formulas

|Metric|Formula|Interpretation|
|---|---|---|
|CV (Cost Variance)|EV − AC|Positive = under budget|
|SV (Schedule Variance)|EV − PV|Positive = ahead of schedule|
|CPI|EV / AC|>1 = efficient|
|SPI|EV / PV|>1 = ahead|

## ⭐ Q. Explain the Scope of Software Metrics `3M` `PYQ`

### Definition

The **scope of software metrics** covers all quantitative measurements applied to software products, processes, and projects to support quality improvement and project management.

### Scope Areas

|Scope Area|What It Measures|Example Metric|
|---|---|---|

|**Reliability**|Failure frequency|MTBF (Mean Time Between Failures)|
|**Maintainability**|Ease of modification|Cyclomatic complexity|
|**Schedule**|Time performance|Schedule Variance (SV)|

### Three Types of Metrics

|Type|Focus|Examples|
|---|---|---|
|**Product**|Software properties|Size, complexity, reliability|
|**Process**|Development process|Review effectiveness, error rates|
|**Project**|PM performance|Cost variance, schedule variance|

## ⭐ Q. Discuss Importance of Risk Management in Project Procurement `3M` `PYQ`

### Definition

**Risk Management** is the process of identifying, analyzing, responding to, and monitoring potential threats to project success.

### Importance

|Importance|Explanation|
|---|---|
|**Early warning**|Identifies threats before they become crises|
|**Reduces failures**|Proactive mitigation prevents small risks from escalating|
|**Protects budget**|Contingency reserves set for cost overruns|
|**Ensures vendor reliability**|Systematic vendor evaluation reduces dependency risk|
|**Improves planning accuracy**|Risk-adjusted estimates are more realistic|
|**Supports decisions**|Risk scores guide make-or-buy choices|

### Risk Response Strategies

|Strategy|For|Description|
|---|---|---|
|**Avoid**|Threats|Change plan to eliminate the risk|
|**Transfer**|Threats|Shift to third party (insurance, contracts)|
|**Mitigate**|Threats|Reduce probability or impact|
|**Accept**|Threats|Do nothing; deal with it if occurs|

### Risk Register (Key Output)

|Risk|Probability|Impact|Score|Response|
|---|---|---|---|---|
|Vendor delays|Medium|High|High|Penalty clauses in contract|
|Budget overrun|Low|High|Medium|Contingency reserve|
|Staff turnover|Medium|Medium|Medium|Cross-training, documentation|

### Key Reasons

|Reason|Explanation|
|---|---|
|**Clarity of goals**|Everyone knows what needs to be done|
|**Team coordination**|Tasks synchronized; no duplication or gaps|
|**Stakeholder alignment**|Expectations managed; no last-minute surprises|
|**Early problem detection**|Issues surfaced through regular reporting|
|**Conflict reduction**|Transparent communication reduces misunderstandings|
|**Decision quality**|PM has complete information for decisions|
|**Motivation**|Team feels valued when kept informed|

### Communication Plan Elements

|Element|Description|
|---|---|
|**Who**|Which stakeholders receive information|
|**What**|Type of information (status, risk, decision)|
|**When**|Frequency (daily standup, weekly report)|
|**How**|Channel (email, meeting, dashboard)|
|**Format**|Summary vs. detailed technical report|

### Communication Channels Formula

```
Channels = n(n-1)/2
→ 5 people  = 10 channels
→ 10 people = 45 channels
→ Adding just 1 person to a 10-member team adds 10 new channels!
```
